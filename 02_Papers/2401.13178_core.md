**LLM 에이전트를 “성공/실패”만이 아니라 “얼마나 진행했는지”까지 보는 통합 벤치마크 + 분석 대시보드.**

## 1. 왜 이 논문인가? (문제의식)

### 기존 벤치마크의 한계

| 한계 | 설명 |
|------|------|
| **단일 라운드·완전 관측** | MMLU·GSM8K 기반 “합성” 에이전트 과제는 현실과 거리가 있음 |
| **최종 성공률만** | 과정 중 진행이 안 보임 → 어려운 환경에서는 대부분 0%라 모델 간 차이가 사라짐 |
| **과제·지표 파편화** | 웹/게임/도구/체화형을 한 프레임워크에서 비교하기 어려움 |

### AGENTBOARD가 제안하는 것

- **9개 환경, 1013개 인스턴스** (체화·게임·웹·도구)
- **다중 라운드 + 부분 관측** 전제
- **진행률(progress rate)** 로 중간 성과 측정
- **WandB 기반 분석 패널** (진행률·그라운딩·난이도·하위기술·궤적)

AgentBench는 MMLU와 상관이 강했고, AGENTBOARD는 **추론·코딩 벤치마크**와 상관이 더 잘 맞음 → **계획(planning)이 더 무거운** 벤치마크.

## 3. 통합 에이전트 프레임워크

### 3.1 POMDP로 보는 LLM 에이전트

- 관측 $o_t$, 행동 $a_t$, 목표 $g$, 메모리 $m_t$ (과거 관측·행동 시퀀스)
- **POMDP** $\langle g, \mathcal{S}, \mathcal{A}, \mathcal{O}, \mathcal{T} \rangle$의 특수 경우

$$
p_\pi(\tau) = p(s_0) \prod_{t=0}^{T} \pi(a_t \mid g, s_t, m_t) \, \mathcal{T}(s_{t+1} \mid s_t, a_t)
$$

![식 1](images/2401.13178/fomula%20(1).png)

### 3.2 실제 구현 (Act-only + Sliding Window)

![그림 2 — 반사 에이전트와 프롬프트 구조](images/2401.13178/figure%20(2).png)

| 설계 선택 | 이유 |
|-----------|------|
| **Act-only** (ReAct 아님) | 장기(최대 30턴) 상호작용에서 thought가 문맥을 잡아먹어 ReAct 이점이 불안정 |
| **Sliding window** | 문맥 초과 시 중단(cutoff) 대신 최근 이력만 유지 → 긴 에피소드 가능 |
| **단순 프레임워크** | 모듈형 에이전트 설계는 LLM마다 호환성이 달라 **기본 능력** 비교에 방해 |

![표 2 — 과제별 목표·궤적 예시](images/2401.13178/table%20(2).png)

## 4. 핵심 기여: 진행률 (Progress Rate)

### 왜 성공률만으로는 부족한가?

- **거의 완료** vs **거의 못 함** → 둘 다 “실패”로 동일
- 예: Llama2-13b vs Mistral-7b — 성공률 2.1% vs 3.9% (비슷) → **진행률 18.9% vs 24.6%** (차이 큼)

### 정의

시간 $t$에서 목표 $g$에 대한 진행도 $r_t \in [0,1]$:

$$
r_t = \begin{cases}
r_t^{\mathrm{match}} = \max_{i,\, 0 \le i \le t} f(s_i, g), & \text{연속 상태 비교 가능할 때} \\[0.4em]
r_t^{\mathrm{subgoal}} = \max_{i,\, 0 \le i \le t} \left( \frac{1}{K} \sum_{k=1}^{K} f(s_i, g_k) \right), & \text{그 외 (하위 목표 기반)}
\end{cases}
$$

![식 2](images/2401.13178/fomula%20(2).png)

- **$r_t^{\mathrm{match}}$:** 현재 상태 ↔ 최종 목표 유사도 (PDDL, WebShop 등)
- **$r_t^{\mathrm{subgoal}}$:** 사람이 라벨한 **하위 목표(subgoal)** 달성 비율 (AlfWorld, Jericho, Tool 등)
- 하위 목표는 **정규식 매칭** 등으로 자동 채점, 인간 검증 3단계 (부록 J)

### 하위 목표 예시

> “clean an egg and put it in microwave”  
> → open fridge → take egg → clean at sink → put in microwave

- 동일 하위 목표라도 **우회 경로(detour)** 허용
- ScienceWorld 등 일부(약 5%)는 gold path 단일화를 위해 문제 문구 수정

### 지표 검증 (사람 vs 자동)

![그림 3 — 인간 진행 점수 vs 자동 진행률 상관](images/2401.13178/figure%20(3).png)

- 8개 과제 × 60 궤적, 저자 4명이 {0, 25, 50, 75, 100%}로 평가
- **Pearson 상관 > 0.95** (모든 과제), Fleiss’ κ로 주석자 간 합의도 양호

## 6. 실험 결과 (핵심만)

![표 3 — 모델별 성공률·진행률 (핵심 표)](images/2401.13178/table%20(3).png)

### 6.1 진행률이 성공률보다 “구분력”이 있다

| 관찰 | 예시 |
|------|------|
| 성공률 비슷해도 진행률은 다름 | Llama2-13b vs Mistral-7b |
| 성공률에 민감한 함정 | CodeLlama-34b가 `finish` 미생성 → 진행률↑ 성공률↓ |
| Embodied/Game에서 open-weight 차이 | 성공률은 다 0%에 가깝지만 진행률로 차이 |

### 6.2 모델 계층 (대략)

1. **GPT-4** — 평균 진행률 **~70%**, 전 하위기술 1위  
2. **Claude, Gemini** 등 proprietary  
3. **오픈 가중치:** Llama3-70b, DeepSeek-67b, xLAM-70b(성공률 20.4%, GPT-3.5-Turbo 상회)  
4. **코딩 LLM 유리** — CodeLlama-34b > Llama2-70b (진행률 +6.2%p)  
5. **에이전트 튜닝** — AgentLM-70b (AlfWorld/WebShop 궤적만으로 학습해도 타 과제 향상)

### 6.3 분석적 평가 (툴킷)

![표 4 — 그라운딩 정확도](images/2401.13178/table%20(4).png)

![표 5 — easy vs hard 성능](images/2401.13178/table%20(5).png)

| 분석 | 요지 |
|------|------|
| **그라운딩** | 유효 행동 비율. 지시 튜닝만으로 에이전트 성능이 안 오를 수 있음 (Vicuna 그라운딩은 높은데 전체 성능은 낮음) |
| **난이도** | hard 예제에서 전 모델 급락 → **조합적(compositional)** 과제가 어려움 |
| **장거리 상호작용** | GPT-4/Claude는 30스텝까지 진행; open-weight는 ~6스텝 후 정체 (그림 4) |
| **하위기술** | memory, planning, world modeling, self-reflection, grounding, spatial navigation (그림 5) |

### 예상 질문 & 답변 포인트

- **Q: AgentBench랑 뭐가 다른가?**  
  A: planning-heavy, 진행률 지표, WandB 분석 패널, 9환경 통일 텍스트 인터페이스.

- **Q: 진행률이 사람 주관 아닌가?**  
  A: 사용자 연구로 상관 0.95+. 다만 subgoal 설계·검증 비용은 한계.

- **Q: 우리 연구/프로젝트에 어떻게 쓸 수 있나?**  
  A: 에이전트 개선 시 success만이 아니라 progress로 ablation; 실패 케이스 궤적 분석으로 약점(그라운딩 vs 계획) 분리.

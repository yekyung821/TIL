# AgentBoard

## 용어 정리

| 용어 | 뜻 | 설명 | 예시 |
|------|---------|----------------|------|
| **다중 라운드** | 챗봇에 질문 **한 번** 하고 끝이 아니라, 환경과 **여러 번** 주고받는 것 | 매 턴: (지금 보이는 것) → (행동 선택) → (결과·피드백) → 다시 반복 | AlfWorld에서 방을 돌며 물건 찾기, WebShop에서 검색·클릭·구매 |
| **부분 관측** | 맵 전체·정답이 처음부터 다 보이지 않고, **조금씩만** 보이는 것 | “전체 상태”를 모른 채 관측만으로 행동 → **탐색·기억**이 필요 (수학적으로는 POMDP) | BabyAI는 시야 안만 보임, Jericho는 게임 규칙을 플레이하며 알아냄 |

**왜 둘 다 강조하나?**  
많은 기존 벤치마크는 “문제 전체가 프롬프트에 다 적혀 있고, 답만 한 번 내면 끝”인 **단일 라운드·완전 관측**에 가깝다. (MMLU 같은 지식 시험, GSM8K 같은 수학 문제를 에이전트 형태로 바꾼 “합성” 과제 등.)  
반면 실제 에이전트(로봇, 웹 비서, 게임 AI)는 **여러 단계를 거치면서**, **아직 모르는 정보를 찾아가며** 일한다. AgentBoard는 9개 환경 모두를 **텍스트**로 통일하면서, 이 두 조건을 만족하도록 사람이 설계·검증했다.

## §1 Introduction

### 1.1 배경 — 왜 LLM 에이전트를 평가하나?

- **범용 에이전트:** 집안, 웹, 게임, 도구 등 **다양한 환경**에서 스스로 목표를 잡고 행동하는 AI. 오래전부터 AI의 큰 목표로 여겨져 왔다.
- **LLM(대규모 언어 모델):** ChatGPT 같은 모델. 글을 이해하고, 지시를 따르고, 여러 문장에 걸친 **계획**을 세울 수 있어서, “에이전트의 두뇌” 후보로 주목받는다.
- 그래서 **“이 LLM이 에이전트로서 얼마나 잘 하는가?”**를 재는 벤치마크가 필요하다. 잘 재야 모델을 고르고, 개선 방향을 잡을 수 있다.

### 1.2 기존 평가의 문제 — “3가지 과제 조건 부족 + 1가지 지표 한계”

| 요구 | 기존 벤치마크에서 흔한 모습 | 왜 문제인가 |
|------|---------------------------|-------------|
| **과제 다양성** | 웹만, 도구만 등 **한 종류**에 치우침 | 체화(집안)·게임·웹·도구를 **한 기준**으로 비교하기 어렵다 |
| **다중 라운드** | 질문 하나에 답 하나인 **QA형**이 많음 | 실제 에이전트는 **여러 번** 시도·수정·탐색한다 |
| **부분 관측** | 문제 전체가 이미 주어진 **합성** 과제 | “돌아다니며 찾기”를 요구하는 현실과 다르다 |
| **지표** | 맨 마지막 **성공/실패(성공률)** 만 봄 | 중간까지 잘 했는지 안 보임. 어려운 과제에서는 대부분 0%라 **모델 차이가 안 보인다** |

### 1.3 AGENTBOARD가 제안하는 것

- **9개 과제, 1013개 환경:** embodied(집/몸), game, web, tool 네 갈래.
- 사람이 각 환경을 **다중 라운드 + 부분 관측**이 되도록 맞추고 검증했다.
- **진행률(progress rate):** “끝까지 성공했는가”뿐 아니라 **중간에 얼마나 왔는가**를 0~1로 잰다.
- **오픈소스 툴킷 + WandB 패널:** 점수만이 아니라 그라운딩·난이도·궤적 등을 **눈으로** 보며 분석한다.

### 2.1 통합 다중 라운드 반사 에이전트

9개 환경이 제각각 프로그램이어도, **에이전트가 보는 입출력 형식**은 같다. 그래서 “통합 프레임워크”라고 부른다.

#### POMDP 정식화

**POMDP(Partially Observable Markov Decision Process)** = “부분적으로만 보이는 세계에서, 순서대로 행동하는 문제”를 수학으로 적은 것이다. 강화학습·로봇 분야에서 오래 쓰인다.

| 기호 | 의미 |
|------|-------------------|
| $g$ | **목표** (예: “계란을 전자레인지에 넣어라”) |
| $\mathcal{S}$ | 환경의 **진짜 상태** (에이전트가 다 보지는 못함) |
| $\mathcal{A}$ | 할 수 있는 **행동** 목록 |
| $\mathcal{O}$ | 에이전트가 **실제로 받는 관측·피드백** (텍스트 설명 등) |
| $\mathcal{T}$ | 행동을 하면 **다음 상태로 어떻게 바뀌는지** (환경 규칙) |
| $m_t$ | **메모리** — 지금까지의 관측·행동 기록 (프롬프트에 넣는 “지난 대화”) |
| $\pi$ | **정책** = “지금 상황에서 어떤 행동을 할지”를 정하는 규칙. 여기서는 **LLM + 프롬프트**가 $\pi$ 역할 |
| $\tau$ | **궤적(trajectory)** — 한 에피소드 동안의 상태·행동 **전체 줄기** |

**궤적이 나올 확률**을 적은 식이다:

$$
p_\pi(\tau) = p(s_0) \prod_{t=0}^{T} \pi(a_t \mid g, s_t, m_t) \, \mathcal{T}(s_{t+1} \mid s_t, a_t)
$$

**설명:**

1. $p(s_0)$: **시작 상태**가 어떻게 잡히는지.
2. $\pi(a_t \mid g, s_t, m_t)$: **목표 $g$**, (이론상) 상태 $s_t$, **지금까지 기억 $m_t$**를 보고 LLM이 **행동 $a_t$**를 고른다.
3. $\mathcal{T}(s_{t+1} \mid s_t, a_t)$: 그 행동을 환경이 받아들이면 **세계가 다음 상태 $s_{t+1}$로 바뀐다** (결정적 전이).
4. $t=0$부터 $T$까지 곱하면 → **한 판(play) 전체 줄기 $\tau$**가 만들어진다.

**의미**  
**순차적 의사결정 문제**를 강조하는 것. 따라서 **진행률**도 “상태가 목표에 얼마나 가까워졌는가”로 정의한다.

#### 통합 프레임워크 — 매 턴에 실제로 일어나는 일

| 단계 | 기호 | 무슨 일이 일어나나 |
|------|------|-------------------|
| 1. 관측 | $o_t$ | 환경이 **텍스트**로 알려줌: 지금 방 설명, 웹 페이지 내용, 오류 메시지 등 |
| 2. 행동 | $a_t$ | LLM이 **텍스트 행동**을 냄: `go to kitchen`, `click[Buy]`, API 호출 문자열 등 |
| 3. 전이 | $\mathcal{T}$ | 시뮬레이터/게임 엔진이 행동을 실행하고 **내부 상태 $s_{t+1}$**로 업데이트 (**결정적**: 같은 행동 → 같은 결과) |
| 4. 피드백 | $o_t = f(s_t, a_t)$ | 바뀐 상태를 다시 **사람이 읽을 수 있는 문장**으로 돌려줌 |

**피드백 $f$의 종류 (논문 예시):**

1. **도움 행동** — `check valid actions`처럼 부르면, 지금 할 수 있는 행동 **목록**을 보여 줌 (그라운딩 실패 줄이기).
2. **성공 행동** — 유효한 행동이면 **상태 변화 설명**을 반환 (예: “You pick up the egg”).
3. **실패 행동** — 행동 문법이 틀리거나 불가능하면 **오류 메시지** (다시 시도하게 함).

**한 줄 요약:** 모든 과제가 **“텍스트로 보고 → 텍스트로 행동 → 텍스트 피드백”** 루프로 같다. 그래서 같은 에이전트 코드·같은 평가 파이프라인을 쓸 수 있다.

#### 구현 선택 — 왜 이렇게 단순하게 했나

![그림 2](images/2401.13178/figure%20(2).png)

*▲ **그림 2:** (왼쪽) 환경과 **반사(reflexive) 에이전트** 루프: 관측 → 메모리에 쌓기 → LLM → 행동 → 환경. (오른쪽) **Act-only 프롬프트** 구조: 목표·과거 대화·지금 관측을 넣고 **행동만** 생성 (ReAct처럼 “생각(Thought)” 단계는 넣지 않음).*

| 선택 | 설명 | 이유 |
|------|-----------|------|
| **Act-only** | “생각 과정” 문장 없이 **행동만** 출력 | 복잡한 에이전트 프레임워크(플래너+실행기 등)를 쓰면 모델마다 호환성이 달라져, **LLM 자체의 에이전트 능력**을 공정히 재기 어렵다 |
| **Sliding window** | 프롬프트가 길어지면 **오래된 턴을 버리고 최근 대화만** 유지 | 문맥 한도를 넘기면 **에피소드 중단**(AgentBench 방식) 대신 **끝까지 플레이**하게 함 |
| **원샷 ICL** | 과제 설명 + **예시 한 개**를 프롬프트에 넣음 | 행동 형식을 맞추기 위한 in-context learning |

### 2.2 세밀한 진행률 (Progress Rate) — 논문의 핵심 지표

#### 왜 성공률만으로는 부족한가

- **성공률:** 제한 턴 $T$ 안에 과제를 **끝까지** 완료한 비율만 센다.
- 문제: “99%까지 했다가 마지막에 실패”와 “첫 행동에서 실패”가 **둘 다 0%**로 같다.
- 어려운 환경에서는 대부분 모델이 0%에 몰려 **순위가 안 갈린다**.
- 환경마다 **보상 점수**는 있지만 스케일이 달라 **벤치마크 전체에서 비교**하기 어렵다.

**진행률**은 “지금까지 중 **가장 목표에 가까웠던 순간**”을 0~1로 기록한다.

#### 정의

각 턴 $t$ 끝에서 $r_t \in [0,1]$:

$$
r_t = \begin{cases}
r_t^{\mathrm{match}} = \max_{i \le t} f(s_i, g), & \text{상태를 목표와 직접 비교할 수 있을 때} \\[0.3em]
r_t^{\mathrm{subgoal}} = \max_{i \le t} \frac{1}{K}\sum_{k=1}^{K} f(s_i, g_k), & \text{하위 목표 } g_1,\ldots,g_K \text{ 로 쪼갤 때}
\end{cases}
$$

*▲ **식 2:** 진행률 정의. $\max_{i \le t}$는 “지금까지 지나온 상태 중 **최고 점수**를 유지”한다는 뜻 (한번 올라간 진행은 안 떨어짐).*

| 유형 | 쓰는 곳 | $f$가 하는 일 |
|------|---------|----------------|
| $r^{\mathrm{match}}$ | PDDL, WebShop, Sheet 등 | **현재 상태**와 **최종 목표 상태**가 얼마나 비슷한지 (연속 점수) |
| $r^{\mathrm{subgoal}}$ | AlfWorld, ScienceWorld, BabyAI, Jericho, Tool 등 | 사람이 적어 둔 **중간 체크포인트** $g_1,\ldots,g_K$를 달성했는지 (보통 0 또는 1) |

#### 하위 목표(subgoal) 예시

**최종 목표:** `clean an egg and put it in microwave`

**사람이 라벨한 중간 단계:**

```
open the fridge → take egg → clean with sinkbasin → put in microwave
```

- 각 $g_k$는 “이 문장이 관측에 나타나면 그 단계 완료”처럼 **정규식** 등으로 자동 채점.
- **우회(detour) 허용:** 다른 방을 먼저 들렀다 와도, 위 하위 목표만 순서대로 달성하면 진행률은 오른다.
- ScienceWorld 등 일부 문제(전체의 약 5%)만 “정답 경로가 하나”가 되도록 문구를 손봤다.

#### 성공률과 함께 쓴다

- **성공률:** 끝까지 했는가 (엄격).
- **진행률:** 얼마나 가까이 갔는가 (완화·구분력 ↑).  
→ §4에서 “성공률은 비슷한데 진행률은 다른” 모델을 구분하는 데 쓴다 (팀원 발표).

### 3.1 환경별 요약

각 환경은 **원래 연구용 시뮬레이터**를 AgentBoard 규칙(텍스트 IO, 다중 라운드, 진행률)에 맞게 **손본 것**이다. 자세한 수치·문장 예시는 **부록 K**.

#### Embodied — “집·실험실 안에서 하는 일”

| 환경 | 무엇을 하나 | AgentBoard에서 한 일 |

### 3.2 주석 검증과 지표 정당화

진행률은 **사람이 적은 하위 목표**에 의존한다. “자동 점수가 사람 눈과 맞나?”를 보여 주는 절이다.

#### 데이터 품질

**3단계 검증**

1. **자기 점검** — 주석 작성자가 담당 데이터 전부 다시 확인.
2. **표본 검사** — 다른 주석자 2명이 과제마다 5~10개씩 골라 검토.
3. **교차 검토** — 여러 과제를 아는 사람이 전 과제 샘플 재검토.

![그림 8](images/2401.13178/figure%20(8).png)

*▲ **그림 8:** 주석자가 시뮬레이터를 **직접 플레이**하며, 각 단계에서 “이 관측이 나오면 subgoal 1 완료”처럼 **라벨을 다는 UI**. 진행률이 허공의 숫자가 아니라, 사람이 검증한 체크리스트에서 나온다는 근거.*

#### 진행률 vs 인간 평가

**방법 :**

- GPT-4 등 3종 모델이 만든 **궤적(플레이 기록)** 480개(8과제×60)를 뽑음.
- 저자 4명이 **자동 점수를 보지 않고**, “이 플레이는 목표의 몇 %까지 왔나?”를 **0 / 25 / 50 / 75 / 100%** 중에서 고름.
- 4명 평균 = **인간 점수**. 이것과 **자동 진행률**이 얼마나 비슷한지 **상관계수(Pearson)** 계산.

**결과:** 모든 과제에서 상관 **> 0.95** → 자동 진행률이 사람이 느끼는 “얼마나 왔는지”와 거의 같다. Fleiss’ κ는 **주석자들끼리 의견이 얼마나 맞는지**를 나타내는 지표.

![그림 3](images/2401.13178/figure%20(3).png)

*▲ **그림 3:** 과제별 **인간 진행 점수 vs 자동 진행률** 산점도·상관. 점이 대각선에 가깝게 모일수록 자동 지표가 타당하다는 뜻.*

### 부록 F — 왜 Act + Sliding window?

**질문:** ReAct(생각+행동)나 요약 메모리가 더 좋지 않나?

![표 9](images/2401.13178/table%20(9).png)

*▲ **표 9:** 같은 모델에 **Act** vs **ReAct** 프롬프트를 썼을 때 과제별 점수. ReAct가 **항상 이기지 않음** — 긴 에피소드에서 “생각” 문장이 **문맥을 차지**해 오히려 불리할 수 있다.*

- **ReAct:** Thought + Action을 번갈아 출력. 계획은 드러나지만 **토큰 소모** 큼.
- **Cutoff:** 문맥 초과 시 **게임 오버** (AgentBench 방식).
- **Summary:** LLM으로 지난 대화를 **요약**해 넣음 — 요약 잘하는 모델이 유리해져 **벤치가 요약 능력까지 재게 됨**.
- **Sliding window (채택):** **최근 턴만** 유지. 짧은 문맥 모델에서 특히 유리 (표 10).

![표 10](images/2401.13178/table%20(10).png)

*▲ **표 10:** GPT-3.5(4k) vs Mistral(32k)에서 **메모리 전략**별 성능. 긴 문맥 모델은 차이가 작고, 짧은 문맥에서는 sliding window가 cutoff보다 낫다는 결과.*

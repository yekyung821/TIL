> - **작성일**: 2026.04.17

> - **Year/Venue**: 2023 / arXiv
> - **Paper**: [arXiv:2305.04091](https://arxiv.org/abs/2305.04091)

> - **한줄평**: "Let's think step by step" 한 문장만으로 생기는 누락/계산 오류를, 계획 수립 단계(Plan)와 해결 단계(Solve) 분리로 줄인 실전형 프롬프트 기법.

---

## 1. 도입 및 배경 (Introduction)
* **논문 요약 및 배경**:
  - Zero-shot-CoT는 간편하지만 추론 중 단계를 건너뛰는 missing-step 오류가 잦음.
  - 계산 실수, 의미 오해가 함께 발생하면 최종 답 정확도가 크게 떨어짐.
  - Plan-and-Solve(PS)는 먼저 문제를 서브태스크로 분해한 뒤 순차적으로 풀도록 유도해 이 문제를 완화함.

* **모델 진화/비교**:

| 비교 항목 | Zero-shot-CoT | Plan-and-Solve (PS/PS+) |
|---|---|---|
| **프롬프트 구조** | 단일 지시 ("step by step") | 계획 수립 + 계획 기반 풀이 |
| **주요 오류** | 누락 단계, 계산 실수 | 누락 단계 감소, 계산 안정성 개선 |
| **사용 난이도** | 매우 쉬움 | 약간 길지만 여전히 간단 |
| **성능 경향** | 태스크별 편차 큼 | 다수 데이터셋에서 일관된 개선 |

* **핵심 개선점/기여도**:

| 핵심 기여 | 설명 |
|---|---|
| **Plan 분리 전략** | 먼저 전체 풀이 계획을 생성해 추론 누락 방지 |
| **PS+ 제안** | 계산 정확도 향상을 위한 상세 지시 추가 |
| **범용 검증** | 10개 데이터셋, 3개 reasoning 문제군에서 일관된 성능 개선 보고 |

## 2. 아키텍처 및 훈련 전략 (Architecture & Prompting)
* **아키텍처/프롬프트 구조**:

| 구성 요소 | 역할 및 특징 |
|---|---|
| **Plan 단계** | 문제를 해결 가능한 하위 단계로 분할 |
| **Solve 단계** | 계획을 따라 단계별 계산/추론 수행 |
| **PS+ 지시문** | 계산 검증, 중간 단계 명시 등 오류 완화 지시 강화 |
| **Zero-shot 사용성** | 예시 없이도 적용 가능하여 실무 도입이 쉬움 |

* **실행 과정**:

| 단계 | 특징 |
|---|---|
| **입력 문제 제시** | 원문 문제를 그대로 입력 |
| **계획 생성** | "먼저 계획을 세워라" 지시로 추론 구조화 |
| **계획 기반 해결** | 각 단계별 풀이 및 최종 답 산출 |
| **검증(PS+)** | 계산/논리 점검 지시 추가 |

> **💡 추가 해설: 왜 성능이 오르는가**
> - CoT는 모델이 "생각은 길게 하되 구조는 보장하지 않음"이 약점임.
> - PS는 구조(계획)를 먼저 고정해 누락 위험을 줄임.
> - 결국 추론 길이보다 추론 구조화가 성능에 더 중요할 수 있음을 보여줌.

## 3. 실험 및 결과 (Experiments)
* **구현 디테일**:

| 항목 | 디테일 |
|---|---|
| **모델** | GPT-3 계열 중심 비교 |
| **평가 범위** | 산술/상식/심볼릭 등 reasoning 문제군 |
| **비교군** | Zero-shot-CoT, Program-of-Thought, 8-shot CoT 등 |

* **벤치마크 결과**:

| 평가 항목 | 결과 요약 |
|---|---|
| **정확도** | 10개 데이터셋 전반에서 Zero-shot-CoT 대비 우세 |
| **강건성** | 누락 단계 오류 감소로 정답률 개선 |
| **효율성** | Few-shot 대비 준비 비용 낮고 성능은 유사 구간 존재 |

* **정성적 결과**:
  - 계획 단계가 생기면서 추론 로그가 더 읽기 쉬워지고 리뷰가 쉬워짐.
  - 실제 팀 스터디/교육에서 "틀린 이유"를 추적하기 좋은 형식임.

* **절제 연구**:
  - Plan 유무, 상세 지시 유무를 분리해 성능 차이를 확인.
  - PS+가 계산/논리 안정성을 추가로 개선함.

## 4. 요약 및 시사점 (Conclusion)
* **요약**:
  - PS/PS+는 모델 파라미터 변경 없이 프롬프트 구조만으로 추론 정확도를 끌어올린 접근.
  - 특히 zero-shot 실무 환경에서 "간편성 + 성능" 균형이 좋음.

* **한계점 (Limitations)**:
  - 프롬프트 길이가 길어져 토큰 비용이 증가할 수 있음.
  - 모델/태스크에 따라 계획이 형식적 문장으로 흐를 위험이 있음.

* **시사점 및 활용 방안**:
  - 에이전트 워크플로우에서 `Plan -> Execute -> Verify` 패턴 설계 근거로 활용 가능.
  - 교육/스터디에서 reasoning trace 품질 관리 기준으로 유용함.

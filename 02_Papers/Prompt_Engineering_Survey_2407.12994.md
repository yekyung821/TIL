# [논문 스터디] A Survey of Prompt Engineering Methods in LLMs for Different NLP Tasks

> - **작성일**: 2026.04.17
> - **Institution**: Independent/Academic Collaboration
> - **Year/Venue**: 2024 / arXiv (Survey)
> - **Paper**: [arXiv:2407.12994](https://arxiv.org/abs/2407.12994)
> - **한줄평**: 최근 2년 프롬프트 기법을 NLP 태스크별로 체계화해, "어떤 문제에 어떤 프롬프트가 맞는지"를 빠르게 판단하게 해주는 실무형 서베이.

---

## 1. 도입 및 배경 (Introduction)
* **논문 요약 및 배경**:
  - LLM 성능은 모델 크기뿐 아니라 프롬프트 설계 품질에 크게 좌우됨.
  - 프롬프트 기법이 폭발적으로 늘면서 태스크별 선택 기준이 혼란스러워짐.
  - 본 서베이는 44편 논문, 39개 프롬프트 방법, 29개 NLP 태스크를 정리해 실무 기준을 제시함.

* **연구 범위/비교**:

| 비교 항목 | 기존 개별 논문 읽기 | 본 서베이 접근 |
|---|---|---|
| **커버리지** | 제한적 | 광범위 (39 methods, 29 tasks) |
| **태스크 연결성** | 낮음 | 태스크-방법 매핑 제공 |
| **실무 활용성** | 개별 해석 필요 | 요약된 선택 가이드 제공 |
| **트렌드 파악** | 어려움 | 최근 2년 흐름 파악 용이 |

* **핵심 기여도**:

| 핵심 기여 | 설명 |
|---|---|
| **프롬프트 분류 체계** | 방법론을 태스크별로 구조화 |
| **성능 정리** | 데이터셋/모델별 보고 성능 종합 |
| **연구 지도 역할** | 후속 실험 설계 시 빠른 출발점 제공 |

## 2. 방법론 정리 프레임 (Taxonomy & Methodology)
* **분류 프레임**:

| 분류 축 | 예시 |
|---|---|
| **추론 강화형** | CoT, Self-Consistency, Tree/Graph-of-Thought |
| **도구 결합형** | ReAct, Program-of-Thought, Toolformer 계열 |
| **지식 강화형** | RAG prompting, citation prompting |
| **제어/안전형** | role prompting, instruction hierarchy, critique prompting |

* **분석 방식**:

| 단계 | 특징 |
|---|---|
| **문헌 수집** | 최근 2년 중심 프롬프트 논문 정리 |
| **태스크 매핑** | QA, IE, summarization 등 태스크별 그룹핑 |
| **성능 비교** | 데이터셋/모델 조합별 reported metric 비교 |
| **한계 논의** | 재현성/평가 불일치/비용 문제 지적 |

> **💡 추가 해설: 서베이 논문을 읽는 포인트**
> - "최고 성능"보다 "어떤 조건에서 잘 되는가"를 보는 것이 실무 전환에 더 중요함.
> - 특히 모델/데이터/평가셋이 다르면 결론이 쉽게 바뀔 수 있음.

## 3. 주요 결과 및 인사이트 (Findings)
* **핵심 결과 요약**:

| 관찰 | 해석 |
|---|---|
| **태스크 의존성 큼** | 범용 최고 프롬프트는 드물고 태스크별 최적 기법이 다름 |
| **복합 전략 우세** | 단일 기법보다 retrieval/critique/tool 결합형이 강함 |
| **비용-성능 트레이드오프** | 성능 향상 기법일수록 토큰/호출 비용 증가 경향 |
| **평가 표준 부재** | 논문 간 비교가 공정하지 않은 경우 다수 |

* **실무적 시사점**:
  - 프롬프트 설계는 "모델 고정 + 프롬프트 실험"의 엔지니어링 사이클로 운영해야 함.
  - 오프라인 벤치마크 성능만 보지 말고 latency/비용/안정성 지표를 함께 관리해야 함.

* **한계/주의점**:
  - 서베이는 2024년 시점 스냅샷으로 최신 기법 반영에 시차가 있음.
  - reported metric 기반 비교는 재현 환경 차이를 충분히 반영하지 못할 수 있음.

## 4. 요약 및 시사점 (Conclusion)
* **요약**:
  - 이 논문은 프롬프트 엔지니어링의 방법론을 태스크 중심으로 재배치한 "지도(map)" 역할을 함.
  - 실무에서 빠르게 실험 우선순위를 정하는 데 특히 유용함.

* **한계점 (Limitations)**:
  - 빠르게 진화하는 분야 특성상 최신성 유지가 어렵고, 일부 기법은 후속 연구로 성능이 크게 변동될 수 있음.
  - 모델별 프롬프트 민감도가 달라 동일 결론 일반화에 주의가 필요함.

* **활용 방안**:
  - 사내 프롬프트 플레이북 구성 시 분류 프레임의 뼈대로 활용 가능.
  - A/B 테스트 설계에서 baseline-variant 선정 체크리스트로 활용 가능.
